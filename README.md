## 1. サービス概要

**まよまよステップ** は、やりたいことがあるのに、できないまま時間が過ぎている人のための **決断を助けるツール** です。

「いつかやりたい」と思っていたことを入力すると、AIが **今日できる小さな一歩** を提案します。

このアプリは、「始め方がわからない」という認知的な問題ではなく、 **「それくらいならやってみるか」という心理的なハードルを下げること** を目的としています。

## 2. このアイデアはどこから生まれたか

### 2-1. きっかけとなった体験・感情

前職を退職した後、次の進路に迷い、「やることはあるのに、何から始めればいいか決められず、結局何もできない」状態に陥りました。
そんなとき、AIに相談したことで思考が整理され、自分の状態を言語化することで納得して次の一歩を踏み出すことができました。
この経験から、自分の状態と行動を整理し、「動けないとき」に一歩踏み出せるきっかけを作れるアプリがあれば、同じように悩む人の役に立てると考えました。

---

### 2-2. なぜそれが「気になった」のか

私は慎重な性格で、行動するまでに時間がかかることがあります。「行動したい気持ちはあるのに動けない」という状態は非常に苦しく、過去に自己嫌悪に陥った経験もあります。

ただ、最初の一歩さえ踏み出せれば、その後はゆっくりでも進められることがわかってきました。
そのため、友人に話すような感覚で気軽に一歩を踏み出せるアプリを作りたいと考えました。

---

### 2-3. コンセプトの再確認

開発を進める中で、友人に触ってもらった際に以下の課題が見えてきました。

#### 課題①：入力フォームに何を入力していいのかユーザーがつかみにくい

**問題点**
- 想定：「いつかやりたい」と思っていたことを **1つだけ** 入力
- 実際：「ギターも麻雀も絵もうまくなりたい」（複数のことを列挙）

**対応内容**
- 入力フォームに **「いつかやりたいと思っていたことを1つだけ入力する」** という文言を追加


#### 課題②：文言がターゲットユーザーに刺さりにくい

**問題点**
- 現在の文言だと「やりたいことが色々あって迷っている人」に限定されてしまう
- 本来のターゲットである「やりたいことがあるのに、できないまま時間が過ぎている人」に刺さりにくい

**対応内容**
以下のような表現で、ターゲットユーザーに刺さる文言に修正しました。

**「できないまま」の状態に共感する表現**
   - 「やりたいけどできないまま」


#### 課題③：「相談」ではなく「決断」のニュアンスを強調

**問題点**
- 「相談」という表現だと受け身のイメージになる
- 目指すべき方向性：「挑戦」「やってみたい」というワクワク感

**対応内容**
- 「相談する」 → **「決める」** で、決断のニュアンスを強調
- **まよまよステップは「決断を助けるツール」** であることを明確にしました。

---

### 2-4. このサービスを作ることができた理由

このサービスのコンセプトを形にできたのは、プログラミング学習を通じて得た気づきがあったからです。

#### 「やる気は行動の後からついてくる」という実感

プログラミング学習を始めた当初は、「完璧に理解してから次に進もう」と考えていました。しかし、実際には **まず手を動かして、エラーを出して、解決していく** というサイクルの中で理解が深まっていくことに気づきました。

「やる気が出るまで待つ」のではなく、 **「まず行動することでやる気が生まれる」** という経験が、このサービスのコンセプトである **「心理的なハードルを下げて、小さな一歩を踏み出す」** という設計に直結しています。

このサービスは、 **「行動から始まる」** というプログラミング学習の本質を、過去の自分と同じように悩んでいる人に届けたいという想いで作りました。

## 8. このアプリの懸念点とその対策

### 懸念点①: AIの提案が的外れだと感じられる可能性

**何が問題になりそうか：**
 AIが提案する「今日できる小さな一歩」が、ユーザーの状況や感情に合っていないと感じられる可能性がある
**なぜそう思うか：**
AIは選択肢をもとに提案を生成するが、ユーザーの細かい状況や感情までは完全に把握できない
**そのために考えている対策：**
提案の再生成機能を追加する(「この提案は合わない」と感じたら、別の提案を生成できるようにする)
 提案の前に「あなたの状況を理解しました」という共感の言葉を入れる
 提案の粒度を調整する（「5分だけ」「1つだけ」など、具体的な時間・量を示す）

### 懸念点②： 継続して使ってもらえない可能性

**何が問題になりそうか：**
最初は使ってくれても、数日で使わなくなる可能性がある
**なぜそう思うか：**
 習慣化は難しく、特に「毎日使う」という行動を続けるのはハードルが高い
**そのために考えている対策：**
リマインダー機能を追加する
振り返り機能で達成感を提供する(「今日もやれた」という達成感を視覚的に表現する)

### 懸念点③：選択肢が多すぎて、逆に迷ってしまう可能性

**何が問題になりそうか：**
選択肢が多いことで、かえって使いづらくなる可能性がある
**なぜそう思うか：**
「選択式だから迷わない」というコンセプトなのに、選択肢が多すぎると本末転倒になる
**そのために考えている対策：**
初回利用時にチュートリアルを表示する
選択肢を段階的に表示する

### 10-2. 技術選定の理由

**なぜこの技術を使うのか**

- **Ruby on Rails を選んだ理由**
  - RUNTEQ で学んできたフレームワークで、CRUD 操作や認証機能の実装に慣れている
- **PostgreSQL を選んだ理由**
  - Rails との相性が良く、本番環境でもそのまま使える
- **OpenAI API を選んだ理由**
  - ユーザーの選択内容をもとに、自然で具体的な提案を生成するため  
- **Tailwind CSS を選んだ理由**
  - シンプルで直感的な UI を素早く実装でき、レスポンシブ対応がしやすい


**今回チャレンジしたい点**

- **OpenAI API を使った AI 提案機能の実装**
  プロンプトの設計を工夫し、ユーザーに寄り添った提案を生成する
- **継続的に使ってもらうための UX 設計**
  - 「投稿して終わり」にならないよう、振り返り機能やリマインダー機能を実装し、ユーザーが「使い続けたい」と思える体験を設計する

**不安な点**
 - **OpenAI API のコスト管理**
API の使用量が増えるとコストがかかるため、リクエスト数を制限する仕組みを検討する必要がある
  
- **AI の提案精度**
 ユーザーの選択内容だけでは、細かい状況や感情まで把握できない可能性がある



## 11. 設計上の工夫・意思決定

### 入力フォーム設計の見直し

ユーザーの状態を直感的に取得できるよう、入力フォームの構成を見直しました。

変更前は自由入力中心でしたが、入力のハードルが高くなる課題があったため、選択式を組み合わせた設計に変更しました。

#### 変更後の項目

- やりたいこと（自由入力・必須）
- 今どんな感じ？（選択式・必須）
- 一歩の大きさ（任意）

これにより、ユーザーが考え込まずに入力できるようにしつつ、AIが状況に応じた現実的な提案を生成できるようにしています。

----

### 入力項目削減の判断

当初は「何が邪魔しているか」という項目を設けていましたが、ユーザーに原因分析を求めることで負担になると判断し削除しました。

MVPでは「迷わず行動に移れること」を優先し、入力は最小限にしています。

----

### コンセプト設計

アプリのコンセプトとして「まよまよ（マヨネーズモチーフ）」を導入しました。
 「迷う（まよまよする）」という状態と音の親和性があり、記憶に残りやすい点からこの名前を採用しています。 

また、「気持ちをまろやかにする」「行動のハードルを下げる」というアプリの目的と、マヨネーズの性質（味をまろやかにする）が一致しているため、 コンセプトとして適していると考えました。

----

### セキュリティ上の考慮（MVP）

本MVPでは、AIの出力結果（previous_proposal）をhiddenフィールドで保持し、再提案時にサーバーへ送信しています。 

ただし、この方法ではDevTools等を用いてクライアント側から値の改ざんが可能です。
 本リリースでは、ProposalモデルとしてDBに保存し、サーバー側でデータを管理・参照することで、改ざんリスクを排除する設計に移行予定です。

---


### 結果出力画面のヘルパーとOPENAI生成の使い分け

- OPENAIの生成結果 → 行動のみを出力
- ヘルパー → 選択肢によって短く「今の気分ならこれがちょうどいいよ」など一行のみの返答を出力

ヘルパーは感情的な励ましではなく、ユーザーが迷わず行動を選べるようにするための 「判断補助の一言」 として使用しています。

----------

### アプリの役割の再定義
本アプリは「やる気を上げる」のではなく、意思決定を代替することで行動を促すことを目的としています。

何をやるか迷う → 動けない
やることが決まる → 動ける

この前提から、「考えさせない設計」を重視しました。

そのため、励ましや抽象的な提案ではなく、具体的で即実行できる行動を1つに絞って提示する設計としています。


-----------


## AI提案ロジックの設計

### 行動提案の方針
- 出力は「行動のみ」に限定
-抽象的な入力でも、具体的な行動に変換
- 「対象 + 数量 + 動作」で一意に決まる形にする
- ユーザー状態に応じた分岐

以下の情報をもとに提案内容を調整しています。

- 気持ち（モチベーション状態）
- 行動の負荷（軽さ）

これにより、ユーザーの状態に応じた現実的な一歩を生成します。


-----------

### 再提案機能

- hiddenフィールドで前回の提案を保持
- プロンプトに前回内容を渡す
- 同じ行動を再提案しないよう制御

これにより、「別のアプローチで試したい」というニーズに対応しています。

--------




## 技術的な工夫

### OpenAI API のプロンプト設計
ユーザーの選択内容をもとに、具体的で実行しやすい提案を生成するため、プロンプトの設計を工夫しました。

- ユーザーの状態（感情・時間）を構造化して渡す
- 行動の粒度を制御（5分 / 30分 など）
- 実行可能性を最優先に設計



### 責務分離を意識したクラス設計

- AI生成・フォールバック処理 → AiGenerator
- 入力処理・レスポンス制御 → Controller

ControllerはAIの成功/失敗を意識せず、結果のみ扱う設計にしています。
